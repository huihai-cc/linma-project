ACHIEVEMENT 01 · DEPLOYED

跨平台邮件自动监控与同步机器人

Python · Playwright · GAS · OneDrive · SharePoint · Teams — 三线并行架构，替代人工每日巡查

已落地 2026.05 · 稳定运行中

1课每天需要手动登录 Xone系统 巡查DSP运营邮件纳期信息。 5课将 Outlook 上的依赖的邮件记录 逐条核对写入总表。此外，各担当通过Teams发送的业务联络信息也需要人工汇总整理。 这些重复性工作不仅消耗大量精力，也容易造成信息遗漏。

为彻底解放这部分人工负担，开发了 三线并行自动化机器人： 线路一 每5分钟自动巡查Xone，智能提取纳期信息； 线路二 利用Automate，OneDrive，Python，识别新增邮件并同步至团队总表； 线路三 自动采集Teams联络信息，经Python加工后通过GAS写入总表，实现1课5课TODO自动汇总。 三条线路均通过GAS完成复杂写入，担当无需任何人工干预即可保持数据实时更新。

ACHIEVEMENT 02 · DEPLOYED

全中心合计表自动分发与多维联动汇总

Google Apps Script · 10中心＋第一WEB本部跨中心汇总 · 6大Sheet维度 · 复合键去重 · 一键执行全流程

已落地 2026上半年 · 持续运行

以往每月需要人工将"全中心合计表"中的数据，逐一拆分整理后分别写入 10个中心各自的 Google Sheets， 还要手动更新收益別・チーム別・HDY本部別・月次サマリー等多个维度汇总表。 操作量大、易出错，每次更新均需人员全程参与。

ACHIEVEMENT 03 · RECENT

DSP设定后台数据与设定表自动对比工具落地

6月15日正式落地 · 面向 Amazon DSP / Amazon PVA 的设定差异自动检测机制

已落地 2026.06.15 · 运营优化中

DSP设定后台数据与设定表自动对比功能已于6月15日正式落地。当前适用媒体为 Amazon DSP 与 Amazon PVA。 第一阶段以辅助自我校正及W校正为目标，通过后台数据直接比对设定表， 自动输出差异结果，帮助担当快速锁定异常项目。

以往需要人工逐项检索确认的内容，例如配信除外、金额设定等， 现在可由系统直接生成比对结果。担当仅需在 不超过5分钟 内查看异常项目，即可完成重点风险确认， 从而进一步确保业务质量并降低人工检查负荷。

本次分享不是统一推广AI平台，也不是要求现场人员学习开发， 而是分享本人实际使用后形成的基础方法和风险边界。 目标对象是：不熟悉AI沟通的现场成员、使用CodeBuddy但不知如何下达指令的成员、 以及希望使用AI整理报告、分析问题或修改小工具的成员。

💡 分享方向A：如何与AI沟通
沟通要点：
1. 先说明背景和目标
2. 描述实际结果与预期结果
3. 提供截图、文件、错误信息和操作步骤
4. 第一次先要求只读分析
5. 明确允许修改与禁止修改范围
安全要求：
6. 要求AI说明风险和验证方法
7. 修改后必须提供验证证据
8. 关键结论必须由人确认
9. AI输出不等于事实
10. 不直接说「帮我修好」
💡 分享方向B：如何把CodeBuddy用到实际工作
✅ 适合的使用场景
  • HTML文字与小范围样式调整
  • Excel公式、宏和错误原因分析
  • 现有Python、GAS、JavaScript脚本检查
  • 报告、手顺和邮件整理
  • 小范围修改
  • 修改前后差异确认
  • 本地验证和完成报告
🚫 禁止或慎用
  • 未明确范围的大规模修改
  • 未备份的重要正式文件
  • 未授权操作真实系统
  • 未确认直接Push或部署
  • 上传账号、客户机密和敏感数据
  • 未经确认运行宏、脚本、删除或批量覆盖
  • 仅凭AI说「已修复」就结束
📋 建议标准操作流程
问题描述 → 提供证据 → AI整理需求 → CodeBuddy只读分析 → 人工确认 → 限定范围修改 → 实际验证 → 保存结果

以下模板可直接复制到ChatGPT或CodeBuddy中使用。替换【】中的内容即可。

模板① 让AI帮我整理问题

📋 复制后替换【】内容即可使用
请帮我把下面的问题整理清楚，暂时不要给解决方案。 【业务名称】 填写业务或工具名称 【当前操作】 1. 2. 3. 【实际结果】 填写现在发生了什么 【预期结果】 填写原本应该出现什么结果 【证据】 填写错误信息、截图、文件名或样本数据 请输出： 1. 问题摘要 2. 已知事实 3. 仍缺少的信息 4. 推荐下一步检查顺序 不要虚构未提供的信息。

模板② 让CodeBuddy先检查，不修改

📋 复制后替换【】内容即可使用
请先只读分析，不要修改任何文件。 【目标文件或目录】 填写路径 【问题】 填写具体问题 【实际结果】 填写当前结果 【预期结果】 填写目标结果 请确认： 1. 问题可能出现在哪个文件或函数 2. 最可能的原因 3. 建议修改范围 4. 是否会影响其他功能 5. 修改后应如何验证 限制： - 不修改文件 - 不运行危险脚本 - 不删除或覆盖数据 - 不提交 - 不Push 请输出分析报告，等待我确认后再修改。

模板③ 确认后授权小范围修改

📋 复制后替换【】内容即可使用
根据刚才的分析，授权进行本次修改。 【允许修改】 - 填写允许修改的文件 - 填写允许修改的功能 【禁止修改】 - 其他页面 - 无关功能 - 数据结构 - 未指定文件 - 正式环境 【完成标准】 1. 2. 3. 要求： - 只修改本次问题相关内容 - 修改前确认git status - 修改后说明具体修改位置 - 不提交 - 不Push - 完成后等待人工确认

模板④ 要求CodeBuddy提供验证证据

📋 复制后替换【】内容即可使用
请对刚才的修改进行实际验证。 必须确认： 1. 语法检查是否通过 2. 页面或程序是否能正常运行 3. 原问题是否已复现并确认解决 4. 是否影响其他功能 5. git diff是否仅包含授权范围 6. 是否存在未验证事项 请输出： - 修改文件 - 修改位置 - 修改内容 - 实际执行的验证 - 验证结果 - 剩余风险 不要只回复「应该可以」或「已修复」。

· 没有修改边界

✅ 推荐

请先只读检查，不要修改。 目标文件：yomiko_kanri.html 当前问题： 表格右侧「编辑」「删除」按钮距离页面边缘太近，视觉拥挤。 预期结果： 桌面端按钮右侧保留自然空白； 两个按钮保持同一行； 左侧筛选区域不得变化； 内容过宽时只允许表格区域横向滚动。 请先说明： 1. 需要修改哪些CSS 2. 是否会影响移动端 3. 建议验证方法 等待确认后再修改。

💡 问题越具体、证据越完整、边界越明确，AI输出越稳定。

🧪 实际案例：Excel宏报错的正确提问方式

第一步：现场人员提供

宏文件名称 · 报错截图 · 点击了哪个按钮 · 当前Sheet名称 · 预期操作结果 · 是否允许修改宏

第二步：先让AI判断

请先分析报错原因，不要修改宏。 当前情况： 点击「デバイス別_取り込み」后报错。 宏预期读取Sheet「デバイス別」，但下载文件中的Sheet名称为「デバイス」。 请判断： 1. 是否是Sheet名称不一致导致 2. 是否需要修改宏 3. 能否通过统一下载文件Sheet名称解决 4. 哪种方式风险更低

第三步：人工决定

如果只需统一Sheet名称即可正常运行，应优先修改操作规则，不要为了「让AI做点什么」而修改宏。

该案例强调：AI不一定必须修改文件；正确判断原因比直接动代码更重要；避免无效开发。
